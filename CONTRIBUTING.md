# CONTRIBUTING — 開発規則

laser-os は長期管理する研究ソフトウェアであり、開発の多くを AI 支援で行う。
本ファイルの規則は、人間・AI を問わずすべての変更に適用される。
**AI に開発を依頼する際は、このファイルを必ずコンテキストに含めること。**

## リポジトリポリシー (必須遵守)

1. **コア物理と GUI の分離**
   `cavsim.core` / `cavsim.analysis` / `cavsim.io` は numpy のみに依存し、
   GUI ライブラリ (PySide6, pyqtgraph 等) を import してはならない。
   逆方向 (GUI → core) の依存のみ許可。

2. **物理変更には必ずテスト**
   物理計算 (行列、素子モデル、ビーム伝搬、安定性判定など) に影響する
   すべての変更は、解析解または信頼できる文献値と照合する pytest を
   追加・更新しなければならない。テスト無しの物理変更はマージ禁止。

3. **物理定数・材料値の捏造禁止**
   未知の屈折率・分散係数・熱光学係数などを AI が推定値で埋めてはならない。
   不明な値は不明として扱い、ユーザーに出典の提供を求めること。

4. **材料データには source_status を付与**
   材料データベース (将来モジュール) の各エントリは
   `source_status: "literature" | "measured" | "estimated" | "unknown"`
   と出典情報を必須フィールドとして持つ。`estimated` / `unknown` の値を
   シミュレーションに使う場合は警告を出すこと。

5. **動作中の挙動の保全**
   既存テストがすべて合格する状態を維持する。意図的な挙動変更は
   CHANGELOG.md に記録し、対応するテストを更新すること。

6. **大規模書き換えの禁止**
   ユーザーが明示的に要求しない限り、ファイル全体の書き直し・
   アーキテクチャ変更・パッケージ名変更 (`cavsim` の改名を含む) を
   行ってはならない。変更は最小差分で行う。

7. **機密・実データのコミット禁止 (公開リポジトリ)**
   本リポジトリは公開されている。以下は**いかなる理由でもコミット禁止**:
   研究ログ、実験生データ、測定結果、論文 PDF、個人文書 (就活文書等)、
   アクセストークン・鍵などの秘密情報。
   実データは本リポジトリの外 (別の非公開保管場所) で管理する。
   コードが実データを読む場合は、リポジトリ外のパスを設定で受け取る
   設計とし、サンプルにはダミーデータのみを使う。
   `.gitignore` に防護ルールがあるが、これに頼らず内容で判断すること。
   AI はユーザーがデータファイルの追加を指示した場合でも、本規則に
   抵触する場合は警告し、リポジトリ外での保管を提案すること。

## AI 支援開発の手順

1. 作業前に対象ファイルを実際に読むこと (記憶からの再生成禁止)
2. 変更は最小差分。無関係なリファクタリング・整形を混ぜない
3. 変更後に `python -m pytest tests/ -q` を実行し、結果を報告する
4. 物理式を追加・変更した場合は `docs/PHYSICS.md` に規約を追記する
5. 検証状況が変わった場合は `docs/VALIDATION.md` を更新する
6. バージョンに影響する変更は `CHANGELOG.md` に追記する
7. AI は実施していない検証を「検証済み」と記述してはならない

## コミット規約

- 1 コミット 1 目的。物理変更と GUI 変更は分ける
- メッセージは `module: 要約` 形式 (例: `cavsim.core: add thermal lens element`)

## テスト実行

```bash
pip install -e .          # または pip install numpy pytest (コアのみ)
python -m pytest tests/ -q
```

GUI はヘッドレス CI では起動できないため、GUI 変更時は import 可能性
(`python -c "import cavsim.gui.main_window"`) と実機での手動確認を行う。

## research/ データ層の追加規則 (laser-research-os 設計の統合)

research/ 配下の研究データ (litdb/matdb/optdb/expdb/compare) を扱う作業は、
上記に加えて以下に従う。出典は `research/DATA_SCHEMA.md` と `docs/DECISIONS.md`。

### セッション開始プロトコル
研究データに触れる作業の前に、まず関連ファイルを実際に読む:
README → 該当タスク → `research/DATA_SCHEMA.md` (データに触る場合) →
`docs/DECISIONS.md` (規約・設計に触る場合)。
チャット履歴や記憶を真実の源にしない。ファイルと記憶が食い違えばファイルが勝つ。

### データ完全性 (絶対規則)
1. **物理値を発明しない。** 「典型値」「およそ」「記憶している係数」を
   データに書かない。不明な値は `value: null`。
2. すべての物理量は PhysicalValue オブジェクト (value/unit/condition/
   source/confidence/data_level) で表す (DATA_SCHEMA §2)。
3. **AI は一次情報源になれない。** AI が訓練データから出した数値は
   最大でも `data_level: extracted` / `source.type: ai_model` /
   `confidence: low` で入り、人間の検証フラグ必須。`verified` 禁止。
4. `verified` への昇格は、検証可能な出典 (DOI＋ページ/表、データシート、
   実測＋実験ログ) を伴って、人間のみが行う。
5. 単位は DATA_SCHEMA §1.3 のとおり明示。無断の単位変換禁止
   (変換は `source.derivation` に記録)。

### 変更管理 (サイレント変更の禁止)
スキーマ・物理規約・ディレクトリ構成・スコープを変える場合:
変更前に ADR ドラフト (動機・変更内容・代替案・影響ファイル一覧) を提示し、
人間の承認を待つ。承認後、DECISIONS.md (status: accepted) → 該当ファイル →
CHANGELOG.md の順で更新。DECISIONS.md / DATA_SCHEMA.md / 実データが
食い違う状態を残さない。

### 禁止事項 (ハードストップ)
- 出典オブジェクトなしの数値を research/ のデータに書く
- 人間の指示なく任意のレコードを `verified` にする
- DECISIONS.md の既存 ADR を編集する (追記専用。supersede で対応)
- 実験ログ・生データを削除する (`deprecated` にする)
- 論文 PDF・測定生データ・個人文書を research/ にコミットする (規則7)
